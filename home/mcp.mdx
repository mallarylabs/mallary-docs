Mallary MCP is a remote social media MCP server for AI assistants, agent workflows, and developer tools that speak Model Context Protocol.

If you want an AI agent to post to social media, upload media, check analytics, list connected platforms, manage webhooks, or inspect publishing jobs without building a custom REST client first, Mallary MCP gives you a structured tool layer over the Mallary platform.

## Add to Cursor

1. Open Cursor Settings
2. Go to Tools & MCP
3. Click + Add new MCP server
4. Paste the following configuration:
```json
{
  "mcpServers": {
    "mallary": {
      "url": "https://mallary.ai/mcp"
    }
  }
}
```
5. Save and go back to Tools & MCP
6. When prompted, click Connect to complete the OAuth flow — you'll enter your Mallary API key in the browser

## Add to Claude Code

1. For Claude Code, run:
```bash
claude mcp add --transport http mallary https://mallary.ai/mcp
```
2. Then run `/mcp` and select Authenticate for Mallary to complete the OAuth flow.

## Add to Claude

Connect directly from Claude without manually configuring an API key:

1. Go to Settings → Connectors
2. Click Add custom connector, enter Mallary as the name and https://mallary.ai/mcp as the URL
3. Click Authorize when prompted to complete the OAuth flow

## Add to Codex
```bash
codex mcp add sentry --url https://mallary.ai/mcp
```
Launch Codex with `codex`. You'll be prompted to authenticate via OAuth.

## Add to VSCode

Open the Command Palette (`CMD+Shift+P`) → `MCP: Add Server` and enter:
```
https://mallary.ai/mcp
```
or, add the following to your VSCode dedicated MCP configuration file `mcp.json`:
```json mcp.json
{
	"servers": {
		"mallary": {
			"url": "https://mallary.ai/mcp"
		}
	}
}
```

## Add to OpenCode

Add to your `opencode.jsonc`:

```json opencode.jsonc
{
  "$schema": "https://opencode.ai/config.json",
  "mcp": {
    "mallary": {
      "type": "remote",
      "url": "https://mallary.ai/mcp",
      "oauth": {}
    }
  }
}
```
Then authenticate:
```bash
opencode mcp auth mallary
```

## What Mallary MCP does

Mallary MCP exposes the same public publishing surface as the Mallary API through MCP tools.

With Mallary MCP you can:

- create upload URLs
- create and schedule social media posts
- inspect job status
- list grouped posts
- delete pending posts
- fetch analytics
- list connected platforms across Mallary's supported platform set
- manage webhooks
- disconnect platforms

## Why teams use Mallary MCP

Mallary MCP is useful when you want to connect social publishing directly into:

- AI assistants
- autonomous agent workflows
- IDE agents
- internal AI tooling
- LLM-based operator workflows

Instead of teaching a model how to compose raw HTTP requests for every task, you can give it a tool surface with clearer actions and safer inputs.

## Base URL

```text
https://mallary.ai
```

## Endpoint

Mallary serves MCP at:

```text
/mcp
```

Supported methods:

- `POST /mcp`
- `GET /mcp`
- `DELETE /mcp`

## Authentication

Authenticate with your Mallary API key:

```http
Authorization: Bearer <mallary_api_key>
```

Identity is resolved from the API key owner.

## MCP transport and behavior

Mallary exposes a remote MCP server over Streamable HTTP for public API parity.

That means:

- MCP tools route through the existing `/api/v1/*` handlers
- posting logic is not duplicated in a separate MCP-only implementation
- tool responses may be returned as event-stream payloads
- `mallary_create_post` runs preflight first by default

## Relationship to the Mallary API

Mallary MCP is not a separate product surface with different business rules.

It uses the same underlying platform as the dashboard, REST API, and CLI, which means:

- plan gating still applies
- platform rules still apply
- account identity still comes from the same Mallary user account
- job and analytics behavior stay aligned across product surfaces

When an MCP tool creates posts, any platform-specific posting behavior still comes from the API request model, especially `platform_options`.

The same applies to scheduling: `mallary_create_post` accepts an absolute `scheduled_at`, or a local `scheduled_at` paired with `scheduled_timezone` using an IANA timezone such as `America/New_York`.

For the exact supported fields and examples by platform, see:

[Platform options reference](/api-reference/endpoint/create#body-platform-options)

The same is true for media validation.

Examples:

- Threads supports text-only posts, one image, one video, or a 2 to 20 item carousel using supported JPG/JPEG/PNG/WEBP/MP4/MOV media.
- YouTube requires exactly one video.
- TikTok video posts require one video, and TikTok photo posts support up to 35 JPEG/WebP images.
- Pinterest requires exactly one image or GIF, or exactly one video.

For the full matrix of platform-specific media rules, see:

[Platform-specific media rules](/api-reference/endpoint/create#platform-specific-media-rules)

## Plans and feature access

MCP access is available on paid plans only.

Current product rules are:

- Free: no MCP access
- Starter: MCP enabled
- Pro: MCP enabled
- Business: MCP enabled

Other related capabilities such as analytics or AI auto replies still follow the plan rules of the authenticated account.

## Error handling

Mallary MCP returns structured errors when the underlying API call is not successful.

Non-2xx results are surfaced with fields such as:

- `http_status`
- `code`
- `message`
- `details` when available

This makes it easier for agents and integrations to decide whether to retry, ask for missing data, or report a plan or validation issue.
